# THE FOUR GOLDEN RULES OF CODING — NON-NEGOTIABLE
# ═══════════════════════════════════════════════════════════════════════════════

These rules apply to ALL agents, ALL tasks, ALL times. No exceptions. No excuses.

## Rule 1: E2E Tests Required (Consolidated First)
- Tests must cover happy path, edge cases, and error states
- **NO MOCKING** of the actual feature being tested
- Prefer adding cases to existing consolidated flows before creating new spec files
- Reuse shared helpers and fixtures to avoid duplicate setup/import flows
- New spec files are allowed only when the scenario cannot fit an existing consolidated flow without harming clarity
- Debug-only or ad hoc regression specs do not belong in the main `bun run test:e2e` suite unless they have a clear permanent purpose
- Normal E2E tests should target 10 seconds or less; anything above 20 seconds requires an explicit documented exception at the test site

## Rule 2: Review Tests
ALWAYS review and critique the tests you create:
- Are all acceptance criteria from the issue covered?
- Are edge cases tested?
- Would these tests catch a regression if the feature broke?

## Rule 3: Run All Tests Before Commit
You can NEVER commit without running and verifying ALL tests including E2E
- **ALL must pass with exit code 0**

## Rule 4: Review Code Before Commit
You can NEVER commit without reviewing and critiquing your own code:
- Does it implement the full spec/issue?
- Is it well-structured and maintainable?
- Are there any hardcoded values that should be configurable?
- Would you approve this in a code review?

# ISSUE-DRIVEN HARNESS WORKFLOW
# ═══════════════════════════════════════════════════════════════════════════════

Use the repository scripts instead of manually editing labels whenever possible:

```bash
./scripts/gh-priority-order.sh
./scripts/gh-priority-order.sh --json
./scripts/gh-issue-view.sh <issue-number>
./scripts/gh-issue-start.sh <issue-number>
./scripts/gh-issue-complete.sh <issue-number> --check-only
```

Normal flow:

1. Start the issue with `./scripts/gh-issue-start.sh <issue-number>`.
2. Read the issue, comments, and rewrite references in `docs/aiq-rewrite.md`, `docs/rewrite-plan.md`, and `docs/rewrite-issues.md` when relevant.
3. Implement the smallest complete slice that satisfies the issue.
4. Add or update tests at the level required by the issue.
5. Run the relevant quality gates.
6. Commit with an issue reference.
7. Open a PR that closes the issue.
8. Request GitHub Copilot and cubic.dev review, wait at least 10 minutes, and inspect feedback before merge.
9. After merge, run `./scripts/gh-issue-complete.sh <issue-number>` so dependent issues can unblock.

Only one issue may be `S-InProgress` at a time. If an issue is already in progress, continue that issue first; use `gh-issue-complete.sh` to finish it or `gh-issue-switch.sh` when explicitly switching context. Do not manually mark a second issue in progress.

Do not close an issue until its checklist is complete or explicitly updated to reflect a scoped follow-up.

`./scripts/gh-priority-order.sh --json` is the machine-readable queue contract used by automation. Keep its `version`, `issues`, `readyIssues`, `nextIssue`, `blockedIssues`, and `inProgress` fields stable unless the harness is updated at the same time.

## GitHub Delivery

Agents may create branches, commit, push, and open PRs for assigned issues. Every PR must request GitHub Copilot review and cubic.dev review, then wait at least 10 minutes before continuing toward merge.

Merging requires passing CI, completion of the Copilot/cubic.dev review wait, and no unresolved blocking review feedback. If follow-up commits materially change the PR after AI reviews, repeat the review wait before merge.

## Hard-Cut Product Policy

- Optimize for a single canonical current-state implementation, not compatibility with historical local states.
- Do not preserve or introduce compatibility bridges, migration shims, fallback paths, adapters, or dual behavior for old local states unless explicitly requested.
- Prefer:
  - one canonical current-state codepath
  - fail-fast diagnostics
  - explicit recovery steps
- Avoid:
  - automatic migration
  - compatibility glue
  - silent fallbacks
  - temporary second paths
- If temporary migration or compatibility code is introduced for debugging or a narrowly scoped transition, it must be called out in the same diff with:
  - why it exists
  - why the canonical path is insufficient
  - exact deletion criteria
  - the ADR or task tracking its removal
- Default stance across the app: remove old-state compatibility code rather than carrying it forward.
